# Session — 2026-04-30

**Ideas touched**: [[mounted-agent-model]], [[mounted-agent-synthesis]] (new)

## What Was Explored

Simon brought five angles for differentiating mounted agents from existing agents — for product page positioning. The angles: open-imprint inverted economics, portability across harnesses, customizability/adaptive behavior, the moat in an open-data model, and builder incentives vs. existing marketplaces. The session ran each thread for non-obvious extensions and pressure-tested where the framings had gaps. Output captured in `product/tokenrip/mounted-agent-synthesis.md`.

## What Emerged

- **The unifying spine: decomposition, not relocation.** All five angles are downstream of a single architectural decision — pulling cognition / context / execution apart so each lives where it belongs. The product page should lead with the architecture; everything else is a consequence. Same lesson as the naming session — name the inversion, not a metaphor.

- **Mounted agents are structurally more capable, not just cheaper.** Hosted agents are capped at "what we can afford per query." Mounted agents have no cap. The capability ceiling difference is a category-grade differentiator that the existing model doc undersold. Worth its own line on the product page.

- **Mounted agents are deflation-friendly; hosted agents are deflation-victims.** Every model price drop makes mounted agents better for free; hosted agents face pricing pressure. This inverts the usual relationship between platform and infrastructure costs.

- **Cross-organization agents only exist in this architecture.** Internal company tools (git-as-agent-OS) cannot be cross-org by definition. Mounted agents are the only architecture where a "Garry Tan agent" or "ResearchGPT for biotech" works. Cross-org agent commerce is not a feature — it is a market that does not exist without mounted agents.

- **Post-company existence is a hidden buyer pitch.** Custom GPTs die when OpenAI kills the product. Mounted agents outlive their builders — the imprint and memory persist on Tokenrip and can be forked. This is a sales argument to *buyers*: you are not betting on the builder being around in 3 years.

- **The harness contributes capability.** Same imprint, different hands depending on where it is mounted. Engineering imprint in Claude Code = git access; in Slack-harness = channel access. Infrastructure-agnostic agents — a category that did not exist before. Should be on the product page as a feature, not implicit.

- **Moat shape changes from protective to attractive.** Software moats defend code (protective). Mounted-agent moats compound usage (attractive). Five concrete moats identified: shared memory (network-effect data), tooling surface (infrastructure), authorship (reputation), inter-agent connections (eBay moat), and observability (positioning moat for regulated industries). The "open data is a feature" framing is true but undersold — it is actually a *market segment* (regulated industries) that mounted agents uniquely serve.

- **The imprint is a behavioral contract.** Today's agents drift; mounted agents are versioned and reproducible. Buying the agent = buying a contract about its behavior, frozen at a version. This reframes the product from "session-grade" to "relationship-grade."

- **Mounted agents are a new programmable interface shape.** Between APIs (return data) and LLMs (reason over text). Reasoning endpoints with state and personality. The API of the agent era. Other software will start calling mounted agents the way it calls APIs today, and Tokenrip is the registry for that interface category.

- **Structural user-builder alignment is the underlying economic insight.** Both sides can leave: user can switch harnesses, builder owns memory. So both are incentivized to make each other happy. Compare to SaaS where the relationship extracts from one or both sides. Mounted agents are the first agent architecture with structural alignment — and this is the answer to "won't builders just leave?"

- **Resolved (proposed): imprint privacy is optional, but the audit log of tool calls is always public to the operator.** This preserves the trust pitch (behavior is auditable) while accommodating builders who want IP protection (prompt can be private). Needs to be locked before the product page ships.

- **Open questions to resolve before product page ships:** BYO-model UX floor (enforce minimum models for certain imprints), shared-memory right-to-deletion architecture, audience-specific harness storytelling (consumers see ChatGPT; developers see "any harness"), fork semantics (forks portable, shared memory stays with canonical imprint).
