# Session — 2026-04-25

**Ideas touched**: [[mounted-agent-model]]

## What Was Explored

Simon brought the engagement agent architecture and asked to pull on the pattern of centralized brain / local harness. The session evolved from "both operators running the same agent" into a comprehensive model for agent deployment, BYO model economics, tiered tooling surfaces, and a potential platform business built on hosting agent intelligence.

## What Emerged

- **Three-layer separation (brain/memory/harness)** — the bootloader pattern already separated brain from harness. Adding memory to Tokenrip completes the separation. The agent becomes a persistent, location-independent entity.
- **BYO model inverts AI economics** — the company provides intelligence and tools, the user provides compute. Company's marginal cost per user is storage, not inference. The unit economics problem that kills AI startups disappears.
- **Charge for hands, not thinking** — tiered tooling surface (free: basic primitives; enterprise: semantic search, webhooks, scheduled ops) is a pricing model with no precedent in AI but natural upgrade triggers and zero user resentment.
- **Five new interaction types** — agent-mediated products (no UI), knowledge-as-a-service (semantic search), progressive knowledge products (improve through use), tool-augmented experiences, and inter-agent tool sharing.
- **Memory ownership is a novel problem** — three models explored (commons, partitioned, layered). Layered model (shared knowledge + private context) is the most architecturally sound.
- **"Build an agent" skill as deployment + distribution** — a skill that walks operators through publishing brain, memory, and provisioning tools. Every deployed agent is a product demo.
- **Token efficiency as natural selection** — in BYO model, bloated brains cost the user directly. Lean beats bloated. A healthy competitive dynamic.
- **AWS analogy but without compute** — Tokenrip provides S3 + API Gateway + IAM + CloudFormation, but not Lambda. The user brings Lambda (their model). Complementary to model providers, not competitive.
