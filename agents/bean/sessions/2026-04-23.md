# Session — 2026-04-23

**Ideas touched**: [[distribution-experiment-framework]]

## What Was Explored

Simon brought a braindump of distribution channel ideas (Twitter, Reddit, influencers, LinkedIn, Moltbook, recipes, Resend playbook) and wanted help organizing them into a structured experiment plan. The session evolved from cataloguing channels into designing a specific distribution workflow that runs entirely on Tokenrip.

## What Emerged

- **Two categories of distribution** — Tokenrip-native (product IS the distribution) vs. standard SaaS (marketing about the product). Category A compounds, Category B is linear. This distinction reframed the entire planning exercise.
- **Three-layer architecture for distribution** — experience-driven content as the hook (Approach B), community channels as the push mechanism (Approach C), and recipes as the conversion tail for people who want to replicate (Approach A). Not competing approaches — they layer.
- **"Content Amplifier" workflow designed end-to-end** — scout tweets with artifact potential → agent drafts actionable artifact (skill/template/prompt) → Alek reviews via Tokenrip thread → publish + post. The full loop runs on Tokenrip. Workflow 2 of three planned experiments.
- **Actionable artifacts over deep analysis** — Simon pushed back on "structured breakdown" assets (Type A) because attention spans on Twitter are too short. Actionable artifacts (skills, templates) win because using them requires CLI install. The value-add and the install funnel are the same action.
- **Alek as first non-technical power user** — his inability to think in systems isn't a liability, it's a signal generator. The workflow scaffolds his role to editorial judgment (ship/fix/kill) while agents handle production. His friction = product roadmap.
- **Distribution experiments ARE the product's first workflows** — the system built to run experiments becomes publishable content (recipes). The experiment and the content it produces are the same thing. Triple win: distribution happens, product improves, workflow recipes emerge.
- **Recipe progression pattern** — start with simplest base case (morning brief v1: agent publishes, you read at a link), let users pull for more complexity (v2: feedback, v3: checklists, v4: team). Ship v1, let demand drive the rest.
- **Git analogy for Tokenrip** — the primitives (CLI, API, threads, assets) are like Git. What's missing is the opinionated workflow — the "Git Flow" equivalent. Recipes are workflow opinions, not documentation.
