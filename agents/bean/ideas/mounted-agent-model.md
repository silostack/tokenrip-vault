# Mounted Agent Model

**Status**: developing
**Created**: 2026-04-25
**Last touched**: 2026-04-30

## Thesis

Mounted agents have three separable layers — imprint (instructions/methodology), memory (accumulated context), and harness (local execution environment). When the imprint and memory live on a shared substrate like Tokenrip and the harness is a thin local client, the agent becomes a persistent, location-independent entity that any operator can mount into any compatible runtime. This inverts AI economics: the user brings their own model (and pays for inference), the company provides the intelligence layer and tooling surface (and pays only for storage/API calls). The company charges for "hands" (tools), not "thinking" (compute).

## Evolution

- **2026-04-25**: Originated from examining the engagement agent's architecture — six Tokenrip assets, two collections, 20-line bootloader, two operators. Recognized that the bootloader pattern (Series 1 blog content) had already separated the imprint. Adding memory to Tokenrip separates all three layers. Explored the BYO model economics inversion (company doesn't pay for inference, user does), five new interaction types (agent-mediated products, knowledge-as-a-service, progressive knowledge products, tool-augmented experiences, inter-agent tool sharing), and the tiered tooling surface as a business model (free = basic primitives, enterprise = semantic search + webhooks + scheduled ops). Identified three memory ownership models (commons, partitioned, layered) with the layered model as likely default. Drew the AWS analogy (S3 = collections, Lambda = user's model, API Gateway = tooling surface). Proposed a "build an agent" skill for deployment.
- **2026-04-27**: The architecture got its first user-facing product. Pressure-test of Alek's marketplace plan (v2) revealed that persona-platform positioning answered the wrong question — it validated distribution, not infrastructure. Path C synthesis (persona = marketing, infrastructure = substance) led to hero/lure architecture: Chief of Staff (Yoda-class, exercising every layer) as the substance, one persona (Garry Tan) as the wave-riding lure. The Chief of Staff is the first agent that uses imprint + private memory + shared knowledge layer + structured intake simultaneously — the validation product for the architecture itself. See [[chief-of-staff-agent]] for the specific product. Key new insight: onboarding-as-feature is the moat, not a feature. Substantive intake produces persistent state on separable memory — portable across agents and surfaces, impossible for Custom GPTs to replicate.
- **2026-04-28**: Locked the public vocabulary. Internal "cloud agent" framing was generic (every AI agent runs in the cloud in 2026) and partially taken (Salesforce Cloud Agents, Lyzr-style headless agents). Landed on **mounted agents** as the category name — architectural-pattern naming that mirrors successful precedents (containerized applications, hosted services, managed services, distributed systems, headless CMS). Engineers parse it instantly via the filesystem-mount metaphor; vocabulary tree extends naturally (mount/unmount, mount point, registry, shared mount, auto-mount). Considered "mountable" but past-participle form follows the established tech-category lineage where `-ed` describes architectural state ("containerized" = built for container deployment, not currently running in one), exactly the convention to inherit. The thing-that-gets-mounted is the **imprint** — chosen over "soul" (vocabulary breaks in technical contexts: "soul registry" / "soul versioning" feel wrong; OpenClaw-coined and adopting it puts Tokenrip on a competitor's shelf), "manifest" (close runner-up; less identity weight; connotes config more than character), and "image" (overloaded — Docker images, ML model images, disk images all collide). Imprint borrows from the publishing industry: Tokenrip is the publisher, each agent is an imprint with its own editorial voice, imprints can be forked, sub-published, retired. Three-layer architecture restated: imprint (instructions, methodology) + memory (accumulated context) + harness (local execution). Same architecture, sharper public vocabulary.

- **2026-04-30**: Synthesis pass for product page positioning — captured in `product/tokenrip/mounted-agent-synthesis.md`. The unifying spine: *decomposition, not relocation.* All differentiation flows from pulling cognition / context / execution apart. Lead with architecture, let consequences fall out. Key non-obvious extensions surfaced beyond what the model doc covered: (1) the capability ceiling lifts — mounted agents are structurally *more capable* than hosted agents at the same price point because there is no per-query cost ceiling for the builder; (2) deflation-friendly — model price drops are free upgrades, not pricing fights; (3) cross-organization agents only exist in this architecture (internal git-as-agent-OS patterns are blocked by IP/security); (4) post-company existence — imprints outlive their builders, hidden sales argument to *buyers*; (5) the harness contributes capability — same brain, different hands depending on mount environment, infrastructure-agnostic agents as a new category; (6) moat shape changes from protective (defend code) to attractive (compound usage); five concrete moats are shared memory, tooling surface, authorship, inter-agent connections, and observability; (7) open data is a market segment (regulated industries pay a premium for explainable AI), not just a feature; (8) the imprint is a behavioral contract — versioned and reproducible, reframes the product from session-grade to relationship-grade; (9) mounted agents are a new programmable interface shape, between APIs and LLMs; (10) user-builder relationship is structurally aligned — both sides can leave, both are incentivized to make each other happy. Proposed resolution to imprint privacy: optional for builders, but tool-call audit log always public to the operator. Five open questions raised that need resolution before product page ships (imprint privacy hill-or-default, BYO model UX floor, shared memory deletion rights, audience-specific harness storytelling, fork semantics).

## Key Challenges Surfaced

- **Memory ownership is novel** — no precedent for shared, multi-contributor agent memory. Open-source solved code ownership (licenses), Wikipedia solved knowledge ownership (commons). Nobody has solved agent memory. The layered model (shared knowledge + private context) is the best candidate but needs design work.
- **Quality control for published imprints** — a poorly-built imprint using Tokenrip tools creates a bad experience that users blame on the platform. May need a reputation/rating system.
- **Model-specific instructions** — an imprint written for Claude may not work for GPT. Unclear whether builders publish model-specific variants or if the imprint format is model-agnostic.
- **Security of tool calls** — a malicious imprint could exfiltrate private operator data through tool calls. Need sandboxing or capability restrictions.

## Open Questions

- What's the actual cost per agent per month for free-tier tooling? At what scale does free become unsustainable?
- How does memory partitioning work at the collection level — separate collections per operator, or row-level ACL?
- Should the "build an agent" skill be a V1 priority (distribution mechanism) or wait until the tooling surface is richer?
- Does the tiered tooling model work if builders can host imprints elsewhere and only use Tokenrip for tools? Is that fragmentation or acceptable usage?
- What does the inter-agent tool registry look like? How does Agent A discover Agent B's capabilities?
- IP model for published imprints — forkable? Licensed? Open by default?

## Non-Obvious Connections

- **Docker for agent intelligence, but alive** — Docker separated applications from infrastructure (static containers). This separates agent intelligence from execution (living entities that evolve through use). The "container" grows smarter.
- **The engagement agent was the proof of concept without knowing it** — the engagement agent architecture accidentally demonstrated the mounted agent model. The generalization was latent in the specific implementation.
- **Token efficiency as competitive pressure** — in the BYO model, bloated imprints cost the user directly. Users switch to leaner imprints. This creates a natural selection pressure for efficient instructions that doesn't exist in any other AI product model.
- **AWS positioning without the compute** — Tokenrip provides all the AWS services EXCEPT Lambda. The user brings their own compute. This is complementary to model providers (every model benefits from Tokenrip) rather than competitive.
- **Nervous system, not brain or muscles** — Tokenrip's position in the stack: model providers are the brain, compute is the muscles, Tokenrip is the nervous system connecting them. The connections that let capabilities find each other and compose.
- **"Build an agent" skill as viral deployment** — every agent that builds itself on Tokenrip is simultaneously a product demo, a data source, and a distribution vector. The deployment mechanism IS the growth mechanism.
